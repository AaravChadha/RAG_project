# Project: Bajaj Capital Mutual Fund Research Chatbot — Pilot Build

## Context

I'm building an AI chatbot for Bajaj Capital Limited's Relationship Managers
(RMs) to query their internal monthly mutual fund research reports. This is a
sanctioned internal Bajaj project. Current goal is a 2–5 RM pilot to validate
usefulness before scaling to 300 RMs.

I'm working solo on this. I'm in the planning/foundation stage — I want to get
the architecture right before writing the chat UI.

## Source Data

Bajaj Capital's research team publishes a monthly circular ("Research
Recommended List of MF Schemes") on the 6th–10th of each month. The May 2026
edition lists 90 schemes across 18 categories (Multi Cap, Flexi Cap, Large Cap,
Large & Mid Cap, Focused, Mid Cap, Small Cap, Value/Contra, Thematic, Dividend
Yield, International, Gold, Balanced Advantage, Conservative Hybrid, Aggressive
Hybrid, Multi Asset Allocation, Arbitrage, Income Plus Arbitrage). The circular
mentions "123 schemes" but only 90 links came through in the email — the
remaining ~33 are likely debt funds in a separate circular I need to track down.

Each scheme has its own PDF report at a predictable URL pattern:
  https://research-host.example/Recommended/<Scheme Name>.pdf

The host is publicly accessible (no auth, no VPN — verified). Spaces in scheme
names become %20. URLs stay constant month-to-month; only the file content
changes when the new month's reports are published.

I have a master CSV (`schemes_master.csv`) with 3 columns — `category`,
`scheme`, `url` — covering all 90 schemes. This is the seed for the schemes
table.

## Document Structure

Every report is generated by **Finalyca** (a fund analytics SaaS) with Bajaj
Capital branding. The template is uniform across all categories — equity,
hybrid, arbitrage, multi-asset, gold, international all share the same section
order and field labels. I've verified this against three samples (Canara
Robeco Multi Cap, ABSL Arbitrage, DSP Multi Asset Allocation).

Reports are 12–19 pages and contain:
- Header block: Benchmark, Inception Date, Min Investment, Expense Ratio, Exit
  Load, Fund AUM, Age, As-Of date, Overview paragraph
- Fund Manager bios (1–3 managers, with Equity/Debt role, qualification, experience)
- Trailing Returns table (1M/3M/6M/1Y/2Y/3Y/5Y/10Y/Since Inception, fund vs benchmark)
- Risk Analysis (1Y and 3Y: Std Dev, Sharpe, Beta, R-Square, Treynor,
  Information Ratio, Up/Down Capture, Tracking Error, Sortino)
- Sector Weights (% by sector, ~15–20 sectors)
- Top Holdings (top 10 with weights, "as on" date)
- Portfolio Characteristics (Total Securities, Avg/Median Mkt Cap, P/E, P/B,
  Dividend Yield, Modified Duration)
- Composition (Equity/Cash/Debt/Derivative/Alternate/Others — varies by fund type)
- Drawdown Analysis (drawdown %, duration days, peak/valley/recovery dates)
- Risk Rating breakdown (credit ratings for debt: A1+, AAA, AA+, Sovereign;
  Equity for stocks)
- Market Cap Composition (Large/Mid/Small Cap %)
- Investment Style matrix (Blend/Growth/Value × Large/Mid/Small Cap)
- Recent Activity: Increase/Decrease in Exposure, New Entrants, Complete Exits
- Performance Trend (rankings vs peer set)
- Rolling Returns (1Y, 3Y, 5Y — min/median/max + observation buckets)
- Month-on-Month returns (last 12 months + MTD)
- Financial Year returns (FY 10 onwards for older funds)
- Calendar Year returns (CY history + YTD)
- Risk & Volatility Measures over time (current, 3M ago, 6M ago, 12M ago snapshots)
- Detailed Portfolio (full holdings table — security name, weight %, sector,
  market cap, instrument type, risk rating, investment style, held since date)
- Sector/Mkt Cap/Style historical trend charts
- AUM trend chart
- Attribution analysis (alpha generators, detractors, overweight/underweight)
- Bajaj Capital + JTSL disclaimer (last page)

### Variations across fund types
- Equity funds: just Equity + Cash composition
- Arbitrage: adds Debt, Derivatives margin (negative %)
- Multi-asset: adds Alternate (Gold/Silver), Others (foreign equities like NVDA, AMZN)
- Hybrid funds with debt have meaningful Modified Duration (~0.7 to 4)
- Pure equity has Modified Duration of 0 or "0E-9"
- Holdings table mixes instrument types: Equity, Debt, Mutual Fund, Commodity,
  Derivatives, Invit/Reit — discriminator is the `Instrument Type` column

### Parser gotchas
- "NA" everywhere → must convert to NULL (not 0, not empty string)
- "0E-9" appears for effective-zero numerics → cast to float
- Date formats: "04 May 2026" in header, "31 Mar 2026" in holdings → normalize
- Holdings table spans multiple pages → concatenate before parsing
- Same security can appear twice with different forms (e.g., HDFC Bank as both
  Equity and Debt) — these are genuinely different positions, do NOT dedupe
- Some holdings have "Held Since: Na" (cash, derivatives) → NULL
- Fund Ranking is missing for newer funds → nullable

## Architecture Decision: Hybrid (Structured DB + RAG), NOT Pure RAG

Most of the data is tabular and numeric. RAG over tabular data is fragile —
the model misreads numbers from retrieved chunks. Pure RAG also makes
cross-fund comparison ("show me all funds with expense ratio < 1.5%") nearly
impossible.

**Layer 1: Structured database** (SQLite for pilot, Postgres later) for all
numeric and categorical data. ~95% of RM questions hit this.

**Layer 2: Vector store** (ChromaDB local for pilot) for the prose: Overview
paragraphs and Fund Manager bios. ~5% of questions hit this.

**Layer 3: LLM with tool use** decides per-question whether to query the DB,
search the vector store, or both. The LLM does not see raw numbers via RAG
chunks — it queries the DB and gets clean values.

For the pilot LLM: Groq's free tier (`llama-3.3-70b-versatile`,
14,400 req/day — enough for 5 users at ~20 queries/day each). Switch to a paid
tier when usage justifies it.

## Database Schema

```sql
-- Master scheme list, populated once from schemes_master.csv
CREATE TABLE schemes (
    scheme_id        INTEGER PRIMARY KEY,
    scheme_name      TEXT NOT NULL UNIQUE,
    amc              TEXT NOT NULL,
    category         TEXT NOT NULL,
    sub_category     TEXT,                 -- e.g. "Equity: Multi Cap" from PDF header
    amfi_code        TEXT,                 -- enrich later from amfiindia.com
    isin_regular     TEXT,
    isin_direct      TEXT,
    source_url       TEXT NOT NULL,
    created_at       TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- One row per scheme per month — the heart of the system
CREATE TABLE fund_snapshots (
    snapshot_id      INTEGER PRIMARY KEY,
    scheme_id        INTEGER NOT NULL REFERENCES schemes(scheme_id),
    as_of_date       DATE NOT NULL,        -- "As On" date from PDF header
    report_month     TEXT NOT NULL,        -- "2026-05" for filtering

    -- Header block
    benchmark            TEXT,
    inception_date       DATE,
    min_investment       TEXT,
    expense_ratio        REAL,
    exit_load            TEXT,
    fund_aum_cr          REAL,
    fund_age             TEXT,
    overview             TEXT,

    -- Trailing returns (% — 1M/3M/6M absolute, 1Y+ are CAGR)
    return_1m REAL, return_3m REAL, return_6m REAL,
    return_1y REAL, return_2y REAL, return_3y REAL,
    return_5y REAL, return_10y REAL, return_since_inception REAL,

    -- Benchmark returns (parallel set with _bm suffix)
    return_1m_bm REAL, return_3m_bm REAL, return_6m_bm REAL,
    return_1y_bm REAL, return_2y_bm REAL, return_3y_bm REAL,
    return_5y_bm REAL, return_10y_bm REAL, return_since_inception_bm REAL,

    -- Risk metrics 1Y
    std_dev_1y REAL, sharpe_1y REAL, beta_1y REAL, r_square_1y REAL,
    treynor_1y REAL, info_ratio_1y REAL, up_capture_1y REAL,
    down_capture_1y REAL, tracking_error_1y REAL, sortino_1y REAL,

    -- Risk metrics 3Y (same set, _3y suffix)
    std_dev_3y REAL, sharpe_3y REAL, beta_3y REAL, r_square_3y REAL,
    treynor_3y REAL, info_ratio_3y REAL, up_capture_3y REAL,
    down_capture_3y REAL, tracking_error_3y REAL, sortino_3y REAL,

    -- Portfolio characteristics
    total_securities     INTEGER,
    avg_mkt_cap_cr       REAL,
    median_mkt_cap_cr    REAL,
    portfolio_pe         REAL,
    portfolio_pb         REAL,
    portfolio_div_yield  REAL,
    modified_duration    REAL,

    -- Drawdown
    drawdown_pct           REAL,
    drawdown_duration_days INTEGER,
    drawdown_peak_date     DATE,
    drawdown_valley_date   DATE,
    drawdown_recovery_date DATE,

    -- Mkt cap composition
    large_cap_pct REAL,
    mid_cap_pct   REAL,
    small_cap_pct REAL,

    -- Flexible JSON blobs for fund-type-specific data
    composition_json      TEXT,   -- {"Equity":95.93,"Cash":4.07,...}
    sector_weights_json   TEXT,   -- {"Financial Services":28.48,...}
    risk_rating_json      TEXT,   -- {"Equity":95.91,"A1+":17.29,...}
    investment_style_json TEXT,   -- {"Large Cap_Blend":4.52,...}
    monthly_returns_json  TEXT,   -- {"2025-05":4.28,"2025-06":4.53,...}
    fy_returns_json       TEXT,   -- {"FY 24":18.20,"FY 25":10.32,...}
    cy_returns_json       TEXT,   -- {"CY 23":16.30,"CY 24":21.93,...}
    fund_managers_json    TEXT,   -- [{"name":...,"role":...,"qual":...,"exp":...}]

    source_pdf_path TEXT,
    ingested_at     TIMESTAMP DEFAULT CURRENT_TIMESTAMP,

    UNIQUE(scheme_id, report_month)
);
CREATE INDEX idx_snap_month  ON fund_snapshots(report_month);
CREATE INDEX idx_snap_scheme ON fund_snapshots(scheme_id);

-- Full holdings table — separate because cardinality is high
-- (~50–200 holdings per scheme × 90 schemes × 12 months = ~100k+ rows/year)
CREATE TABLE holdings (
    holding_id       INTEGER PRIMARY KEY,
    scheme_id        INTEGER NOT NULL REFERENCES schemes(scheme_id),
    report_month     TEXT NOT NULL,
    security_name    TEXT NOT NULL,
    weight_pct       REAL,
    sector           TEXT,
    market_cap       TEXT,        -- "Large Cap"/"Mid Cap"/"Small Cap"/NULL
    instrument_type  TEXT,        -- "Equity"/"Debt"/"Mutual Fund"/"Commodity"/"Derivatives"/"Invit/Reit"
    risk_rating      TEXT,        -- "EQUITY"/"A1+"/"AAA"/"Sovereign"/NULL
    investment_style TEXT,        -- "Growth"/"Value"/"Blend"/NULL
    held_since       DATE
);
CREATE INDEX idx_hold_scheme_month ON holdings(scheme_id, report_month);
CREATE INDEX idx_hold_security     ON holdings(security_name);

-- Audit log for compliance (every RM query gets logged)
CREATE TABLE query_log (
    query_id         INTEGER PRIMARY KEY,
    user_id          TEXT,
    timestamp        TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    question         TEXT NOT NULL,
    sql_executed     TEXT,
    sources_cited    TEXT,        -- JSON array of (scheme, month, page)
    final_answer     TEXT,
    user_feedback    TEXT,        -- "thumbs_up"/"thumbs_down"/NULL
    feedback_comment TEXT
);
```
